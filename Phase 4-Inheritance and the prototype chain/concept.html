Level 00

What even is a prototype?

In class-based languages like Java or C++, you define a blueprint (a class) and then stamp out objects from it. JavaScript is different — it uses prototypal inheritance. There are no blueprints. Instead, objects inherit directly from other objects.

Think of it like a reference chain. Every object in JavaScript holds a hidden link — called [[Prototype]] — that points to another object. That second object is its "prototype" (its parent). If you ask the object for a property it doesn't have, it silently looks at its parent. And if the parent doesn't have it either, it looks at its parent. This chain of lookups is the prototype chain.

💡
Core mental model

Every object has a hidden [[Prototype]] link to another object (or null). Property lookups walk up this chain until they find the property or hit null.

The simplest possible example

javascript basics 
// Using __proto__ here just for illustration.
// In real code, use Object.create() or class syntax.
 
const animal = {
  breathes: true,
};
 
const dog = {
  __proto__: animal,  // dog's [[Prototype]] = animal
  name: "Rex",
};
 
console.log(dog.name);     // "Rex"    ← own property
console.log(dog.breathes); // true     ← inherited from animal
console.log(dog.size);     // undefined ← not found anywhere in chain

When you access dog.breathes, JavaScript:

  • Checks dog itself — not found
  • Follows the [[Prototype]] link to animal — found! Returns true

When you access dog.size, it checks dog, then animal, then Object.prototype, then hits null — returns undefined.

Level 01

Property lookup & property shadowing

The lookup process walks up the chain. It stops at the first match it finds — even if later ancestors also have that property. This is called property shadowing.

javascript property shadowing 
const o = {
  a: 1,
  b: 2,
  __proto__: {
    b: 3,  // ← b also exists in parent
    c: 4,
  },
};
 
console.log(o.a); // 1  ← own property
console.log(o.b); // 2  ← own property SHADOWS the parent's b:3
console.log(o.c); // 4  ← not on o, found on parent
console.log(o.d); // undefined ← not found anywhere

How the lookup works step-by-step

For o.b:

  • Does o have an own property called b? Yes → return 2. Stop.

For o.c:

  • Does o have an own property called c? No → go up
  • Does o.[[Prototype]] have c? Yes → return 4. Stop.

For o.d:

  • Does o have d? No → go up
  • Does parent have d? No → go up
  • Does Object.prototype have d? No → go up
  • Next is null — chain exhausted → return undefined
✅
Checking own vs inherited

Use Object.hasOwn(obj, 'prop') (modern) or obj.hasOwnProperty('prop') to check if a property belongs to the object itself — not its prototype chain.

javascriptown vs inherited 
Object.hasOwn(o, 'a'); // true  — own property
Object.hasOwn(o, 'c'); // false — inherited from prototype
Level 02

Inheriting methods & the this keyword

JavaScript doesn't have "methods" as a separate concept. A method is just a function stored as a property. When an object inherits a function from its prototype, the function works exactly as if it were the object's own — with one key rule:

💡
The golden rule of this

When an inherited function runs, this refers to the object that called it — not the prototype where the function lives.

javascriptthis with inheritance 
const parent = {
  value: 2,
  method() {
    return this.value + 1; // 'this' = whoever calls this fn
  }
};
 
const child = {
  __proto__: parent,
};
 
console.log(parent.method()); // 3  (this = parent, parent.value = 2)
console.log(child.method());  // 3  (this = child, child.value → parent.value = 2)
 
child.value = 10; // add own 'value' to child
console.log(child.method());  // 11 (this = child, child.value = 10)
// parent.value is still 2 — child shadows it

Notice: method() is defined once on parent, but child inherits and calls it. this inside the function dynamically points to the calling object. This is why shared methods on prototypes are so powerful.

Level 03

Constructor functions & the new keyword

Manually setting __proto__ on every object is tedious. Constructor functions automate this. When you call a function with new, JavaScript does four things behind the scenes:

  • Creates a fresh empty object
  • Sets that object's [[Prototype]] to ConstructorFn.prototype
  • Runs the function with this pointing to the new object
  • Returns the new object (unless the function explicitly returns another object)
javascriptconstructor function 
function Box(value) {
  this.value = value; // 'this' = the new object being created
}
 
// Add shared methods to the prototype ONCE
Box.prototype.getValue = function() {
  return this.value;
};
 
const b1 = new Box(42);
const b2 = new Box(99);
 
console.log(b1.getValue()); // 42
console.log(b2.getValue()); // 99
 
// Both share the SAME function in memory — not duplicated
console.log(b1.getValue === b2.getValue); // true ✓
 
// Verify the prototype chain
Object.getPrototypeOf(b1) === Box.prototype; // true
// b1 → Box.prototype → Object.prototype → null

Why put methods on the prototype?

If you defined getValue inside the constructor with this.getValue = function() {...}, every single instance would get its own copy of that function — wasting memory. Putting it on Box.prototype means all instances share one copy.

⚠️
Don't re-assign prototype

Never do Box.prototype = { ... }. This replaces the entire prototype object and loses the constructor reference. Mutate it with Box.prototype.method = ... instead.

Level 04

Building longer chains (inheritance)

To make one constructor inherit from another — creating a proper class hierarchy — you link their prototypes using Object.setPrototypeOf().

javascriptmulti-level inheritance 
function Animal(name) {
  this.name = name;
}
Animal.prototype.breathe = function() {
  return `${this.name} breathes`;
};
 
function Dog(name, breed) {
  Animal.call(this, name); // call parent constructor
  this.breed = breed;
}
// Link Dog's prototype to Animal's prototype
Object.setPrototypeOf(Dog.prototype, Animal.prototype);
Dog.prototype.bark = function() {
  return `${this.name} says woof!`;
};
 
const rex = new Dog("Rex", "Labrador");
console.log(rex.bark());    // "Rex says woof!"    ← Dog.prototype
console.log(rex.breathe()); // "Rex breathes"      ← Animal.prototype
console.log(rex.toString()); // "[object Object]"  ← Object.prototype

The full prototype chain for rex:

rex → Dog.prototype → Animal.prototype → Object.prototype → null

When you call rex.breathe(), JS walks right along this chain, skipping rex (no own), skipping Dog.prototype (no breathe), and finding it on Animal.prototype.

Level 05

Class syntax — just sugar over prototypes

ES6 class syntax was introduced to make prototype-based inheritance easier to write and read. But here's the key insight: it compiles down to the exact same prototype chain. Classes don't introduce a new inheritance model — they're syntactic sugar.

javascriptclass syntax = prototype chain 
class Animal {
  constructor(name) {
    this.name = name;
  }
  breathe() {
    return `${this.name} breathes`;
  }
}
 
class Dog extends Animal {
  constructor(name, breed) {
    super(name);   // calls Animal's constructor
    this.breed = breed;
  }
  bark() {
    return `${this.name} says woof!`;
  }
}
 
const rex = new Dog("Rex", "Labrador");
 
// The prototype chain is IDENTICAL to Level 4:
// rex → Dog.prototype → Animal.prototype → Object.prototype → null
Object.getPrototypeOf(Dog.prototype) === Animal.prototype; // true
Aspect Constructor function (ES5) Class syntax (ES6+)
Under the hood Prototype chain Same prototype chain ✓
Inheritance Object.setPrototypeOf(Child.prototype, Parent.prototype) class Child extends Parent
Parent constructor Parent.call(this, ...args) super(...args)
Methods Enumerable by default Non-enumerable (better)
Readability Verbose, error-prone Clean, explicit
✅
Practical advice

Use class syntax in modern code — it's cleaner and less error-prone. But now that you understand the prototype chain behind it, you can debug inheritance issues that classes abstract away.

Level 06

All the ways to create & set a prototype

MDN documents several patterns. Here's every one, with tradeoffs:

1 · Object literal (implicit Object.prototype)

javascript 
const obj = { a: 1 };
// obj → Object.prototype → null (automatic)

2 · Object literal with __proto__ key (standard, not deprecated)

javascript 
const parent = { greet() { return "hi"; } };
const child  = { __proto__: parent, name: "Alice" };
// child → parent → Object.prototype → null
// NOTE: { __proto__: x } in a literal is standard + fast.
// obj.__proto__ = x (accessor form) is deprecated — avoid it.

3 · Object.create()

javascript 
const parent = { greet() { return "hi"; } };
const child  = Object.create(parent);
child.name = "Alice";
 
// Bonus: create an object with NO prototype at all
const dict = Object.create(null);
// dict has no toString, no hasOwnProperty — pure key/value store

4 · Constructor function + new

javascript 
function Person(name) { this.name = name; }
const alice = new Person("Alice");
// alice → Person.prototype → Object.prototype → null

5 · class / extends (recommended)

javascript 
class Person { constructor(n) { this.name = n; } }
class Employee extends Person { constructor(n, role) { super(n); this.role = role; } }
const e = new Employee("Alice", "Engineer");
// e → Employee.prototype → Person.prototype → Object.prototype → null

6 · Object.setPrototypeOf() (mutate after creation — avoid when possible)

javascript 
const a = {};
const b = { greet() { return "hi"; } };
Object.setPrototypeOf(a, b);
// Works, but mutating the prototype chain at runtime
// disables JS engine optimizations. Use only if you must.
Level 07

Gotchas & performance tips

Gotcha 1 · Monkey-patching built-ins

Adding methods to Array.prototype, Object.prototype, etc. is called monkey patching. Don't do it — it can break future JS features and surprise other developers.

javascript 
// ❌ Don't do this
Array.prototype.sum = function() {
  return this.reduce((a, b) => a + b, 0);
};
// ✅ Do this instead — a standalone utility function
function sumArray(arr) {
  return arr.reduce((a, b) => a + b, 0);
}

Gotcha 2 · for...in iterates the whole chain

javascript 
const parent = { x: 1 };
const child  = { __proto__: parent, y: 2 };
 
for (const key in child) {
  console.log(key); // "y" then "x" — x comes from parent!
}
 
// Guard with hasOwn to only get own properties:
for (const key in child) {
  if (Object.hasOwn(child, key)) console.log(key); // only "y"
}
 
// Or use Object.keys() — only returns own enumerable properties
Object.keys(child); // ["y"]

Gotcha 3 · Long chains hurt performance

⚠️
Chain length matters

Each step up the prototype chain is an extra lookup. For hot code paths (called thousands of times), a 10-level chain is measurably slower than a 2-level one. Keep chains short — rarely deeper than 3–4 levels. Accessing a non-existent property is worst: it always traverses the entire chain before returning undefined.

Gotcha 4 · Mutating a shared prototype affects all instances

javascript 
function Cat() {}
Cat.prototype.speak = () => "meow";
 
const c1 = new Cat();
const c2 = new Cat();
 
// Mutate the prototype AFTER instances exist
Cat.prototype.speak = () => "ROAR";
 
console.log(c1.speak()); // "ROAR" — c1 is affected!
console.log(c2.speak()); // "ROAR" — c2 is affected too!
// All instances share the same prototype object — changes are global.
